Feedback & Programmes Page

Technical Description

The Feedback page combines two distinct sections: a scrollable programme directory and a validated feedback form. The programme directory uses a fixed-height container with overflow-y: auto and a custom styled scrollbar built with ::-webkit-scrollbar pseudo-elements. Each programme card uses CSS Grid with three columns (thumbnail, text, button) to achieve a consistent horizontal layout. The feedback form is structured using HTML5 <fieldset> and <legend> elements for semantic grouping. JavaScript handles live client-side validation on form submission each required field is tested against a condition function, and error spans are toggled using classList.add/remove('show'). A live character counter on the message textarea updates on every input event and turns red when fewer than 50 characters remain. On successful validation the form is hidden and a success message block is made visible using classList.add('show').

Accessibility

All form inputs have explicitly associated <label> elements linked by matching for and id attributes. Required fields are marked with an asterisk and programmatically indicated via the required attribute. Error messages use role="alert" on the success notification so screen readers announce it automatically. The multiple-select interest list includes an aria-label. The scrollable programme list has an aria-label describing it as a scrollable region. Colour contrast on all form elements meets WCAG AA standards, and focus styles are preserved on inputs and buttons.

Team Page

Technical Description

The Team page uses a CSS Grid layout with three equal columns to display team member cards. Each card is built entirely with CSS no JavaScript is required for the hover interaction. The reveal of the member details panel is achieved using a max-height transition from 0 to 300px combined with an opacity transition, triggered by the :hover and :focus pseudo-classes on the parent .member-block container. The image scaling and lift effect on hover uses transform: scale() and translateY() with CSS transitions. An ::after pseudo-element on the thumbnail provides a linear-gradient colour overlay that grounds the name badge text against any background image. The bottom detail panel uses border-top with the primary amber colour as a visual anchor linking it to the card above.

Accessibility

Each .member-block has tabindex="0" so keyboard users can focus it and trigger the hover state through :focus. Each card carries a descriptive aria-label that identifies the student and their role. The team grid container has an aria-label. Images use meaningful alt text describing each person's role rather than a generic caption. The page heading hierarchy is maintained (h1 for the page banner, h2 would be used for named sections if they existed).

Content Page — Hydro Power

Technical Description

The Hydro Power content page follows the same two-column CSS Grid structure established in the shared template: a sticky sidebar for internal navigation and a main article column. The page uses scroll-margin-top on each section to offset the fixed navigation bar when jumping to anchor links. A data visualisation block uses CSS-only bar charts built with inline width percentages and background colour fills no chart library is required. A callout block highlights key statistics using border-left and a semi-transparent rgba() background. Image captions are implemented as <figcaption> elements inside <figure> wrappers for semantic correctness. The page also includes a related links section to the other three content pages for cross-navigation.

Accessibility

The sidebar is wrapped in <aside> with aria-label="On-page navigation". Heading hierarchy is strict (h1 for page title, h2 for major sections, h3 for subsections). All images have descriptive alt text. Figure captions supplement but do not duplicate alt text. The Go to Top button carries a descriptive aria-label. The CSS-only bar chart includes visually hidden text values using a .sr-only class for screen reader users.

Challenges and Lessons Learned

The most significant challenge was implementing reliable client-side form validation in JavaScript. Early iterations triggered all errors simultaneously on submit, which overwhelmed users visually. The solution was refactoring the validation logic into a declarative array of field objects each containing the field ID, error element ID, and a check function so the validation loop is consistent and easy to extend. This taught me the value of data-driven programming patterns over repetitive conditional blocks. A secondary challenge arose with the Team page hover reveal: using height: auto in a CSS transition does not animate smoothly because the browser cannot interpolate from a fixed value to auto. Switching to max-height with a known upper bound resolved the issue and produced a clean slide-down effect. This deepened my understanding of how browsers compute layout during transitions.

Accessibility Considerations

Across all pages I implemented semantic HTML5 elements (<header>, <main>, <nav>, <aside>, <section>, <fieldset>, <legend>, <figure>, <figcaption>, <footer>) to provide meaningful document structure for assistive technologies. All interactive elements are keyboard accessible form inputs, buttons, and team cards can all be reached and operated without a mouse. Colour contrast meets WCAG AA standards throughout. All images have descriptive alt text conveying meaning relevant to context. ARIA roles and labels are applied where HTML semantics alone are insufficient, such as on the scrollable programme list, the modal success alert, and the team member cards.
