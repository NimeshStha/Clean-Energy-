Home Page

Technical Description

The Home Page uses a full-viewport hero section built with CSS Grid and Flexbox to divide the layout into a text column and an interactive gallery grid. The hero background uses layered CSS radial-gradient() functions animated with a @keyframes pulseGlow rule to create a living, atmospheric backdrop without JavaScript. The card components use CSS transitions (transform: translateY and box-shadow) on hover to create a tactile hover response. The content gallery linking to all four content pages uses CSS clip-path on absolutely positioned overlay divs that reveal on hover via opacity transitions. Global CSS custom properties (CSS variables) ensure consistent colour usage throughout the page.

Accessibility

All images include descriptive alt attributes. The navigation uses a <nav> element with an aria-label. The hero section carries a role="banner" via the <header> element. ARIA labels are provided on interactive gallery items that act as links. Heading hierarchy is maintained (h1 → h2 → h3). Colour contrast between the primary amber (#f4a020) and dark background (#0d1f17) exceeds WCAG AA ratio of 4.5:1.

Gallery Page

Technical Description

The gallery is rendered dynamically using JavaScript. An array of eight gallery item objects is defined in the script, each containing title, image URL, alt text, and description. The forEach method iterates over this array to create <div> elements and inject them into the DOM grid. JavaScript event listeners handle the hover interaction (border outline change via element.style), the click-to-expand modal (classList manipulation to add the .open class), and the close behaviour (click on close button, click outside modal, and Escape key). User customisation — changing the modal background colour and font family — is handled via input and change event listeners that update the modal box's inline styles using the DOM style API.

Accessibility

All gallery cards have tabIndex="0", role="button", and descriptive aria-label attributes for keyboard navigation. The modal has role="dialog" and aria-modal="true". Focus is moved to the close button when the modal opens. The Escape key closes the modal. All images have alt text injected from the data array.

Content Page — Solar Power

Technical Description

The Solar Power content page uses a CSS Grid two-column layout: a sticky sidebar containing an internal navigation menu and a main article area. The sidebar uses position:sticky with a top offset calculated from the navigation bar height CSS variable, ensuring it remains visible as the user scrolls. Sections use scroll-margin-top to account for the fixed header when navigating via anchor links. Images use CSS box-shadow and border-radius for polished presentation. The callout block uses a left border accent achieved with border-left and a semi-transparent background created with rgba() colour values.

Accessibility

The sidebar is wrapped in <aside> with aria-label="On-page navigation". Heading hierarchy is strict (h1 for page title, h2 for major sections, h3 for subsections). All four images have descriptive alt text specific to their content. The Go to Top button uses a descriptive aria-label. Anchor links use meaningful text rather than generic phrases.

Challenges and Lessons Learned

The main challenge was building a shared CSS template that all four team members could use without conflict. Early iterations caused style collisions when students added page-specific rules that overrode global ones. The solution was enforcing CSS specificity discipline — all page-specific rules use class selectors scoped to the page (e.g., .hero for the home page) and are clearly commented as "DO NOT OVERRIDE IN GLOBAL CSS". This taught me the importance of CSS architecture in team environments and the value of naming conventions. A secondary challenge was the JavaScript-rendered gallery — ensuring screen reader accessibility required learning about aria-live regions and focus management, which deepened my understanding of inclusive design beyond visual appearance.

Accessibility Considerations

Across all pages I implemented: semantic HTML5 elements (<header>, <main>, <nav>, <aside>, <article>, <section>, <footer>) to provide meaningful document structure for screen readers. All interactive elements are keyboard accessible. Colour contrast meets WCAG AA standards — the primary amber on dark background achieves a ratio above 4.5:1. All images have descriptive alt text that conveys meaning rather than just describing the image format. ARIA labels supplement interactive elements where HTML semantics alone are insufficient.
