Collaborative submission — Students 1, 2 & 3. This group consists of three members. The Profile page and Sitemap page, which would ordinarily be assigned to a fourth student, were completed as a shared team effort. All three students contributed to the planning, implementation, testing, and documentation of both pages. Individual contributions are identified throughout this report using the tags ST1 ST2 ST3.

User Profile Page

Technical Description

The User Profile page was designed and built jointly by all three team members. The page implements a multi-step profile builder driven entirely by JavaScript — no backend or server is involved. The layout uses a two-column CSS Grid: a controls panel on the left and a profile output panel on the right. ST1 was responsible for the overall page layout, CSS Grid structure, and the visual styling of the step buttons, progress bar, and profile field cards, ensuring the page integrated cleanly with the shared template.

ST2 implemented the core JavaScript logic: a profileData object stores all user responses across three steps; the runStep() function uses the browser's native prompt() API to collect input and writes values back to the object; the updateProgress() function iterates over all nine data keys using Object.values() to calculate completion percentage and updates the progress bar width via inline style. The confirmation banner is revealed via classList.add('show') when all nine fields are populated.

ST3 wrote the profile output section, including the step reveal animation (@keyframes fadeIn combined with classList.add('visible')), the empty state placeholder, and the revisit buttons. ST3 also tested the full three-step user journey across multiple browsers and devices, logging and fixing an issue where cancelled prompts incorrectly cleared previously saved values.

Accessibility

All step buttons have descriptive aria-label attributes. The profile output sections use aria-label on each container div. The confirmation banner uses role="alert" so screen readers announce it automatically when it appears. The controls panel and output panel are within <main>, preserving landmark navigation. The progress bar is accompanied by a visible text percentage label so the information is not conveyed by colour alone. All interactive buttons are keyboard focusable.

Sitemap Page

Technical Description

The Sitemap page presents the full architecture of the LumEn website as an interactive SVG diagram embedded directly in the HTML. The entire diagram is a single inline <svg> element with a fixed viewBox of 0 0 1000 620 and width="100%", making it fully responsive without JavaScript. ST1 planned the site hierarchy, determined the node positions and connector paths, and drew the initial SVG layout — placing nodes at precise coordinates for the five-level tree: Splash → Home → Main Pages → Content/Sub-pages → Editors/Validation.

ST2 implemented the interactive features: each node is wrapped in an SVG <a> element with a class="node-link" and an href pointing to the corresponding page, making the entire map navigable. Arrow connectors use an SVG <marker> element referenced via marker-end="url(#arrow)". CSS hover and focus rules animate the node fill and stroke colours to provide clear interactive feedback. ST2 also added the <title> and <desc> elements on the root SVG for accessibility.

ST3 wrote the legend block at the base of the SVG, distinguishing the three node types (key pages, main pages, sub-pages) by fill colour with labelled colour swatches. ST3 also reviewed the final connector lines against the submitted file list to confirm every page in the project is represented, and corrected a merge conflict in the SVG source that had left duplicate legend markup from two different git branches.

Accessibility

The root <svg> carries role="img" and an aria-label describing the diagram. A <title> and <desc> element inside the SVG provide a text summary for screen readers. Each clickable node has an aria-label on its wrapping <a> element and an inner <title> child for tooltip and AT support. Focus styles are applied via CSS so keyboard users can see which node is active. The page uses a semantic <main> landmark and a descriptive <h1> heading in the banner.

Challenges and Lessons Learned

Coordinating three people on two pages that neither belonged to any single student introduced unique challenges around ownership and consistency. The main technical difficulty on the Profile page was the browser's native prompt() behaviour — cancelling a dialog returns null, which we initially used to overwrite existing saved values. The fix was a simple guard: only update the data object if the returned value is non-null and non-empty, preserving prior answers on revisit. This was identified during ST3's cross-browser testing and patched by ST2.

On the Sitemap, coordinating SVG connector line coordinates across three contributors proved error-prone — small edits to node positions broke connecting lines written by someone else. We resolved this by agreeing on a fixed coordinate grid before any implementation began, treating node centres as named constants in a shared planning document. The git merge conflict in the legend was a direct result of two students editing the same SVG region simultaneously; resolving it reinforced the importance of branch discipline and clear task boundaries even within a small group.

Accessibility Considerations

Across both pages the team implemented semantic HTML5 landmarks (<header>, <main>, <nav>, <footer>) consistent with the shared template. The SVG sitemap is fully accessible to screen readers through role="img", <title>, <desc>, and per-node aria-label attributes. The Profile page progress bar is supplemented by visible text so colour is never the sole carrier of information. All interactive elements — step buttons, revisit buttons, SVG node links — are keyboard focusable and carry meaningful labels. WCAG AA colour contrast is maintained throughout, consistent with the global palette defined in the shared CSS.

Compliance with JANET Regulations

Both pages comply with JANET regulations governing web page publication at UK universities. All content is original and produced collaboratively by the three team members for educational purposes. No copyrighted material is reproduced without attribution. The Profile page collects user input through browser prompt dialogs only — no data is transmitted to any server or stored beyond the current browser session, and no personal data is persisted. The Sitemap page contains no third-party content. No discriminatory, defamatory, or offensive content is present on either page. The submission reflects honest collaborative academic work and does not misrepresent individual or group contributions.
