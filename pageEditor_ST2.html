Splash Screen

Technical Description

The Splash Screen is the user's first point of contact with the LumEn website. It was built as a standalone page that does not use the shared HTML template, although it closely matches the website's agreed colour palette, typography, and tone. The full-screen background uses layered CSS radial-gradient() functions combined with a subtle grid overlay created using the CSS background-image property with repeating linear gradients to produce a structured, atmospheric backdrop. A decorative sun glow element is positioned using position: fixed with a radial-gradient and animated with a @keyframes pulseGlow rule that scales and fades the element continuously, adding visual depth without JavaScript. The CSS loader animation uses a rotating border element created with border-top-color on a circular div, animated via @keyframes spin a pure CSS technique requiring no images or JavaScript. All text elements use CSS animation with @keyframes fadeDown and animation-delay staggering to produce a sequential reveal effect as the page loads.

JavaScript Functionality

Two JavaScript features are present. First, a countdown timer uses setInterval() to decrement a counter variable every 1,000 milliseconds and update the DOM text content of the countdown element, displaying "Entering site in 4…3…2…1" dynamically. Second, the Skip Intro button calls a skipIntro() function that immediately clears the interval timer using clearInterval() to cancel the auto-redirect, then navigates to home.html using window.location.href. The HTML meta refresh tag handles the automatic redirect as a fallback: <meta http-equiv="refresh" content="4;url=home.html">.

Accessibility

The loading ring carries role="status" and an aria-label so screen readers announce it as a loading indicator. Decorative background elements use aria-hidden="true" to prevent them from being announced. The team names section has aria-label="Group members" to provide context. The skip button includes a descriptive aria-label explaining its purpose. All text maintains sufficient contrast against the dark background, exceeding WCAG AA requirements.

Action Impact Simulator

Technical Description

The Action Impact Simulator (ais.html) allows users to select clean energy actions and receive a dynamically calculated impact score. The page uses the shared HTML template and global CSS, with additional embedded CSS handling the action card grid layout, selection states, and the sticky feedback panel. The card grid uses CSS Grid with repeat(auto-fill, minmax(280px, 1fr)) to create a responsive layout that adapts from one to three columns depending on viewport width. Each action card is rendered dynamically by JavaScript using a forEach loop over an array of nine action objects, each containing a title, description, image, category, and impact points value. Cards are inserted into the DOM using innerHTML. The selected card state uses classList.add('selected') to apply a CSS border and background tint, and a CSS ::after pseudo-element displays a checkmark badge. The sticky feedback panel uses position: sticky with bottom: 0, remaining fixed at the bottom of the viewport as the user scrolls through the action cards.

JavaScript Functionality

A JavaScript Set object tracks which action IDs are currently selected. When a card is clicked, a toggle function checks whether the ID is in the Set using has() if so it removes it with delete(), otherwise it adds it with add(). After every toggle, an updateFeedback() function iterates over the selected Set, looks up the corresponding action object, and sums the impact points. Conditional logic (if/else if/else) compares the total against three thresholds Low (0–6), Medium (7–13), and High (14+) and updates the badge text, badge CSS class, feedback message, and page background colour accordingly using document.body.style.background. A reset button clears the Set, removes all .selected classes from cards, and resets the feedback panel to its default state. Keyboard accessibility is implemented by listening for the Enter and Space keys on each card via a keydown event listener.

Accessibility

Each action card has role="checkbox" and aria-checked attributes that are toggled between "true" and "false" by the JavaScript selection logic, providing accurate state information to screen readers. Cards also carry descriptive aria-label attributes including the action title and impact points. The feedback panel uses role="region" and aria-live="polite" so screen readers announce score updates without interrupting the user. All images have meaningful alt text defined in the data array. The reset button has a descriptive aria-label.

Content Page — Wind Energy

Technical Description

The Wind Energy content page follows the same two-column CSS Grid layout established by Student 1's content page template: a 240px sticky sidebar on the left and a main article area on the right. The sidebar uses position: sticky with a top offset derived from the --nav-h CSS variable, keeping navigation accessible as the user scrolls through the long-form content. Each article section carries a scroll-margin-top value to account for the fixed header when users navigate via anchor links in the sidebar. The callout block uses a left border accent (border-left: 4px solid var(--primary)) with a semi-transparent amber background created using rgba() to visually distinguish key statistics from body text. Seven distinct content sections Introduction, How Wind Works, Types of Wind Energy, Benefits, Student Actions, Global Picture, and References ensure sufficient depth for meaningful internal navigation.

Accessibility

The sidebar nav is wrapped in an <aside> element with aria-label="On-page navigation" to distinguish it from the main navigation. All seven images carry descriptive alt text specific to their content context. A visible "Go to Top" button with aria-label is fixed to the bottom-right of the viewport using position: fixed. Heading hierarchy is maintained strictly throughout one h1 for the page title, h2 for major sections, and h3 for subsections supporting both readability and screen reader navigation.

Challenges and Lessons Learned

The most significant challenge in building the Splash Screen was implementing the countdown timer alongside the CSS-based auto-redirect without them conflicting. The meta refresh tag fires independently of JavaScript, so if the skip button cleared the JS timer but the meta tag still redirected, users would be forced to the home page regardless. The solution was to rely on JavaScript's window.location.href for the actual redirect and treat the meta tag purely as a no-JS fallback a distinction that required careful thinking about progressive enhancement. For the Action Impact Simulator, the main challenge was managing the dynamic rendering of cards alongside accessible ARIA state. Initially, aria-checked was not being updated on selection, meaning screen reader users received no feedback when selecting or deselecting cards. Adding setAttribute('aria-checked', 'true/false') inside the toggle function resolved this. This experience taught me that accessibility cannot be bolted on after the fact it must be considered at every stage of JavaScript-driven DOM manipulation.

Accessibility Considerations

Across all pages I implemented, accessibility was approached as a design requirement rather than an afterthought. The Splash Screen uses ARIA roles on the loader and hidden attributes on decorative elements. The Action Impact Simulator uses role="checkbox" and aria-live to keep screen reader users informed of both card selection state and score updates in real time. The Wind Energy content page uses semantic HTML5 sectioning elements throughout <article>, <section>, <aside>, <nav> to provide a meaningful document outline. All interactive elements across my pages are keyboard accessible, tested by navigating with Tab and Enter keys only. Colour contrast on all text elements meets or exceeds the WCAG 2.1 AA standard of 4.5:1 for normal text.
